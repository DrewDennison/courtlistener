REST API Tutorial

The CourtListener API does a lot and can be big and complex. Below are a few example projects to help get you started.

Understanding our Content

CourtListener has four major types of data: audio files, PACER data (via the RECAP project), opinions, and judges. These object types are linked together where possible. For example, judges are linked as relational data from opinions, audio files, and PACER dockets.

The biggest challenge when using these APIs is understanding how to fetch very large objects from it. For example, some of the dockets in our system have over 10,000 entries and some class action cases have thousands of parties. If a single API call could request this many objects from our database, we would struggle to keep up — generating and serving a JSON object with the text of thousands of documents is generally a bad idea! As a result, many of the objects, like dockets, are broken up into paginated API endpoints, each of which lets you get the parts you need, whether that's paginating over docket entries or over a list of parties. We'll give an example of this below.

The other challenge of these APIs is that the data is complex. For example: federal dockets have parties; parties have attorneys which have roles in particular cases; and attorneys work at particular firms. Parties also have criminal complaints against them, those complaints have outcomes, etc. As an API user, you wind up needing to understand both the semantics (What is an "attorney-role"?) and the code (How do I fetch an attorney-role from the API?). None of this is easy, but we'll get into it below.

Beyond these complexities, there are just things you have to know about the legal system. What's the best way to identify a case? What are the components of a docket number? What do jargony terms like per curiam and en banc mean? Some of these questions are beyond our purview, but we'll do our best to answer them when we can.
